# 游戏体验设计师 (UX/UI)

## When to use this skill

- 需要将策划文档转化为可视化 XMind 思维导图
- 梳理界面层级、操作流程与边界状态
- 推导策划案遗漏的空态/满态/异常状态
- 识别交互歧义与 UI 风险点
- 生成验收 CheckList 与三级风险清单

## Instructions

### 第零步：Excel 文件摸底（仅当输入含 .xlsx 时执行）

调用 MCP `large-file-reader` 的 `get_file_summary` 获取所有 Sheet 名称与**实际行数**，
再逐 Sheet 按真实行数读完整，**禁止凭经验估算截断**。

> ⚠️ 跳过此步骤将导致 Sheet 末尾功能模块被截断遗漏。

---

### 第一步：文档分析

1. 读取策划文档，过滤背景故事、纯数值配置、服务端逻辑。
2. 识别：功能目标、涉及界面数量、主操作流程。
3. 定位触发关键词：入口路径、触发条件、特殊状态。

---

### 第二步：内容推导（逐步思考）

1. **梳理流程骨架**：确定主流程步骤顺序。
2. **扫描边界**（Self-Ask 自问）：
   - "如果玩家资源为 0，这个操作会怎样？" → 补充置灰/拦截/引导
   - "如果这个列表有 0 条数据，显示什么？" → 补充空态
   - "如果名字很长，怎么处理？" → 补充截断/跑马灯
   - "操作完成后，界面状态是否明确？" → 补充反馈/刷新
3. **定位歧义**：找出策划案中可被解读为两种以上行为的描述，标记 ❓。
4. **梳理界面**：逐一确认每个界面的触发路径、展示内容和操作控件。

---

### 第三步：输出格式

**唯一格式**：Markdown 缩进列表（`- ` + 2空格缩进），三分支结构：

```
- 需求单名字
  - 规则与核心
    - [一句话说明，≤100字]
    - **⚡(AI推导) 边界：[边界条件]**
  - 流程
    - [操作 → 结果]
      - **⚡(AI推导) [异常/边界状态]**
        - **⚠️(AI推导) [具体风险]**
      - **❓(待确认) [步骤歧义项]**
  - 界面
    - [界面名称]
      - 触发路径：入口A → 动作
      - [元素/控件]：说明
      - **⚡(AI推导) [推导的控件或状态]**
      - **⚠️(AI推导) [风险]**
      - **❓(待确认) [控件歧义项]**
```

**符号系统：**

| 符号 | 含义 | 格式要求 |
|------|------|----------|
| `⚡(AI推导)` | 策划未明确的功能性状态/控件（AI补全） | **必须加粗** |
| `⚠️(AI推导)` | AI识别的显示或体验风险 | **必须加粗** |
| `❓(待确认)` | 策划歧义、矛盾或未定义行为 | **必须加粗** |
| 无标记 | 策划明确说明的内容 | 正常文字 |

**三条硬性规则：**
- **规则A（原子化）**：每个 UI 元素独占一行，并列信息必须拆分为独立子节点。
- **规则B（按钮补全）**：所有 `[按钮]` 节点必须在同行补全点击行为；指向的目标界面**必须**在界面分支中存在。
- **规则C（二级弹窗扫描）**：先枚举所有一级界面，再扫描全部 `[按钮]` 行为，补充二级界面节点。

> ⚠️ ⚡ 和 ⚠️ 节点**只能**标注策划案中未明确说明的内容，禁止对已知内容重复表述。

---

### 第四步：自动保存到桌面

**Step 1：探测桌面路径**（禁止硬编码用户名）

```bash
echo %USERPROFILE%\Desktop   # Windows
echo ~/Desktop               # macOS / Linux
```

**Step 2：生成文件名**

```
{需求单名字}_UX架构图_v{版本号}_{时间戳}.md
```

示例：`门派试炼_UX架构图_v143_20260330_100000.md`

**Step 3：写入文件**（仅写入三分支缩进列表，禁止混入 `##` 标题/表格/CheckList）

**Step 3.5：对话框展示**

文件写入后，在对话框中依次输出：
1. **三级风险优先级清单**（🔴高 / 🟡中 / 🟢低，每级 2–5 条）
2. **三维验收 CheckList**（功能验收 / 数据验收 / UX验收）

> 这两部分不写入文件，仅在对话框展示。

**Step 4：询问是否沉淀示例**

AI 先自评本次输出质量，再提问：
> 「本次输出是否需要更新到 Skill 示例？（AI自评：**建议更新/建议跳过**，原因：[一句话说明]）回复**是**我来帮你更新，回复**否**跳过。」

若回复**是**，则追加到 `references/examples.md`，格式：
```
## 示例N：{需求名} · v{版本号} {简短描述}
> {本次输出的亮点说明}
{完整输出内容}
> 📌 来源：{需求名} · v{版本号} · {日期}
```

## Best practices

1. **描述质量优先**：⚡/⚠️/❓ 内容必须是策划案未明确说明的，宁可不写，不凑字数。
2. **先摸底后读取**：有 Excel 时必须先调 `get_file_summary`，绝不截断。
3. **按钮必须闭环**：规则B是防漏的硬性自检，每个按钮指向的界面都要在界面分支中存在。
4. **文件内容纯净**：`.md` 文件只写缩进列表，避免 XMind 导入出现多余节点。
5. **示例实战化**：每次高质量输出后及时更新到 `references/examples.md`。
